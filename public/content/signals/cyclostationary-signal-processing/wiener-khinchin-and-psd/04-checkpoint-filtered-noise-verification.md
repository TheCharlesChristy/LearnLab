## Checkpoint: filtered noise, verified

Case file, closing this module: everything so far has either been exact algebra or, in the very first checkpoint, a clean deterministic signal. Real data is neither — your actual recordings are finite, noisy realizations of a random process. This checkpoint asks the real question: does Wiener–Khinchin still hold up once randomness and finite data are both in the picture?

:::callout{kind="key"}
Plan: generate white noise, pass it through a simple filter to produce a **filtered (colored) random process** with real autocorrelation structure, then estimate that process's autocorrelation two ways from one finite, noisy realization — directly, and via the periodogram (|DFT|² then inverse-transformed). If Wiener–Khinchin genuinely holds for real, noisy, finite data (not just tidy deterministic examples), the two estimates should still agree.
:::

::widget{type="code-runner" language="python" rows=26 starter="import random\nimport cmath\n\nrandom.seed(42)\nN = 256\n\n# Step 1: generate white noise\nwhite = [random.gauss(0, 1) for _ in range(N)]\n\n# Step 2: filter it (a simple 3-tap FIR low-pass) to produce a colored process\ndef filter_signal(x):\n    y = []\n    for n in range(len(x)):\n        v = 0.5 * x[n] + 0.3 * x[(n - 1) % len(x)] + 0.2 * x[(n - 2) % len(x)]\n        y.append(v)\n    return y\n\ny = filter_signal(white)\n\n# Step 3: direct sample autocorrelation (circular, unnormalized)\ndef direct_autocorr(x, max_lag):\n    N = len(x)\n    R = []\n    for k in range(max_lag):\n        s = 0\n        for n in range(N):\n            s += x[n] * x[(n - k) % N]\n        R.append(s)\n    return R\n\n# Step 4: periodogram-based autocorrelation via DFT/IDFT\ndef dft(x):\n    N = len(x)\n    return [sum(x[n] * cmath.exp(-2j * cmath.pi * k * n / N) for n in range(N)) for k in range(N)]\n\ndef idft(X):\n    N = len(X)\n    return [sum(X[k] * cmath.exp(2j * cmath.pi * k * n / N) for k in range(N)) / N for n in range(N)]\n\nMAX_LAG = 8\nY = dft(y)\npower = [abs(v) ** 2 for v in Y]\nR_via_fft_full = idft(power)\n\nR_direct = direct_autocorr(y, MAX_LAG)\nR_via_fft = [v.real for v in R_via_fft_full[:MAX_LAG]]\n\nprint('Direct autocorrelation:      ', [round(v, 2) for v in R_direct])\nprint('Via periodogram (FFT):       ', [round(v, 2) for v in R_via_fft])\nprint('Max abs difference:          ', round(max(abs(a - b) for a, b in zip(R_direct, R_via_fft)), 8))\n"}

:::reveal{title="What to look for"}
The two rows of numbers should match to within tiny floating-point rounding (the "max abs difference" line should print something like `0.0` or a number many decimal places away from anything else in the output) — even though `y` is genuinely random, filtered noise, not a clean deterministic signal like the first checkpoint. Try changing the filter taps in `filter_signal`, or the noise seed, and re-running: the two columns should keep matching no matter what you change, because the identity holds for *any* signal, random or not, finite or not.
:::

Case log: the Wiener–Khinchin theorem isn't a tidy fact that only works on textbook examples — it holds for the messy, random, finite-length data you'll actually be handed in the field, which is exactly what you just verified with your own hands. You now have the complete first-principles toolkit the roadmap promised: probability, linear algebra, Fourier analysis, stationary random processes, and the autocorrelation/PSD relationship that ties them all together. From here, the case moves to a question you haven't had to face yet: this Wiener–Khinchin machinery still assumes you can estimate a clean PSD from real data in the first place — and a single raw FFT of a noisy recording turns out to be a surprisingly bad way to do that. That's exactly where spectral estimation, and the rest of the roadmap toward cyclostationarity, picks up next.
